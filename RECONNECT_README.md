# WebSocket Reconnection During Active Calls

## Проблема

При переключении VPN или смене сетевого подключения во время активного WebRTC звонка:
- **WebRTC медиа-канал** продолжает работать (проходит через установленные NAT/firewall правила)
- **WebSocket сигнальное соединение** разрывается и переподключается
- При переподключении создается новый WebSocket, но старые данные сессии (статус, кандидат, ICE данные) остаются привязанными к старому ws
- Результат: ошибки "peer not found" при попытке обновить offer/answer

## Решение

Реализован **механизм восстановления сессии** при переподключении WebSocket:

### 🔧 Ключевые изменения:

#### 1. Обновленная логика `ADD_USER`
```javascript
// Проверяем, есть ли уже активная сессия этого пользователя
const existingUserMap = users[userId];
if (existingUserMap && existingUserMap.size > 0) {
    // Находим активную сессию (не idle или с активным звонком)
    for (const [oldWs, userData] of existingUserMap) {
        if (userData.status !== 'idle' || userData.candidate) {
            // Восстанавливаем сессию - сохраняем все данные, только меняем ws
            userData.ws = newWs; // Подменяем WebSocket
            break;
        }
    }
}
```

#### 2. Новые вспомогательные функции
- `getActiveUser(userId)` - получает активного пользователя независимо от WebSocket
- `getUserByWs(userId, ws)` - получает пользователя с fallback на активного

#### 3. Обновленные функции поиска пользователей
Все функции теперь используют `getUserByWs()` вместо прямого `users[userId].get(ws)`:
- `handleOffer` 
- `handleAnswer`
- `handleDecline`
- `handleAddIce`
- `handleSwap`
- `updateStatus`

### 🚀 Как работает:

1. **Пользователь в звонке** (статус: `calling`, `ringing`, `in_call`)
2. **VPN переключается** → WebSocket разрывается
3. **Клиент переподключается** → отправляет `ADD_USER`
4. **Сервер обнаруживает** существующую активную сессию
5. **Восстанавливает сессию**: 
   - Сохраняет все данные (статус, кандидат, ICE, streamIds)
   - Подменяет только WebSocket на новый
   - Уведомляет собеседника о переподключении
6. **Звонок продолжается** без прерывания

### 📋 Логирование

Новые логи для отслеживания переподключений:
```
[RECONNECT DETECTED] User user123 reconnecting with status: in_call
[SESSION RESTORED] User user123 session restored with status: in_call  
[PEER NOTIFIED] Notified user456 about user123 reconnection
```

### 🧪 Тестирование

Создан тестовый файл `test-reconnect.js` для проверки:

```bash
# Запуск сервера
npm run dev

# В другом терминале - запуск теста
node test-reconnect.js
```

**Тест симулирует:**
1. Подключение двух пользователей
2. Установку звонка между ними  
3. Переподключение первого пользователя (имитация VPN)
4. Обновление offer после переподключения
5. Переподключение второго пользователя
6. Обновление answer после переподключения

### 🔍 Мониторинг

В Telegram логах появляются уведомления:
- 🔄 `[RECONNECT] User reconnecting during in_call status`
- ✅ `[SESSION RESTORED] User session restored with status: in_call`

### ⚡ Преимущества

- **Бесшовное переподключение** - звонок не прерывается
- **Сохранение состояния** - все данные сессии остаются
- **Автоматическое восстановление** - не требует действий от пользователя
- **Уведомления собеседника** - партнер знает о переподключении
- **Обратная совместимость** - работает с существующими клиентами

### 🛡️ Безопасность

- Проверка принадлежности сессии по `userId`
- Валидация статуса перед восстановлением
- Очистка старых WebSocket соединений
- Защита от дублирования сессий

## Использование

Клиентский код остается без изменений - механизм работает автоматически при стандартном переподключении через `ADD_USER`.

Для получения уведомлений о переподключении собеседника, обрабатывайте сообщение:
```javascript
ws.on('message', (data) => {
    const message = JSON.parse(data);
    if (message.type === '/peerReconnected') {
        console.log(`Собеседник ${message.name} переподключился`);
        // Можно показать уведомление в UI
    }
});
```
